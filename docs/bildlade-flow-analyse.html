TravelMate Bildlade-Flow Analyse

Stand: aktuelle Repository-Analyse ohne Codeänderung

Ziel: erklären, warum echte Aktivitätsbilder kurz sichtbar sein können und danach wieder durch Kategorie-Platzhalter ersetzt werden.

Kernaussage: Der Bildlade-Flow hat aktuell zwei Wahrheiten: lokale Bilder in ActivityGallery und zentrale Bilder im Trip-State aus useTripWorkspace. Wenn echte Bilder nur lokal ankommen, gehen sie beim Wechsel von Aktivität, Tag oder nach einem späteren Trip-Update wieder verloren. Dann greift die Galerie erneut auf Platzhalter zurück.

1. Wo entsteht der finale Bild-Slide-State?

Der finale Slide-State entsteht im Frontend in:

C:\Users\finnw\Documents\Reiseplaner\frontend\components\ActivityGallery.vue

Die Galerie bekommt von DayOrbitCard.vue eine einzelne TripActivity:

<ActivityGallery
  :key="(selectedActivity || day.activities[0])?.id"
  :activity="selectedActivity || day.activities[0]"
  :city="city"
/>

Intern baut ActivityGallery daraus GallerySlide-Objekte. Ein Slide enthält immer einen Fallback und optional ein echtes Bild:

type GallerySlide = {
  key: string
  image: GalleryImage
  realImage: GalleryImage | null
  fallbackImage: GalleryImage
  label: string
  variant: number
}

Es gibt zwei Fallback-Orte: normale Kategorie-Platzhalter in ActivityGallery.vue und einen zusätzlichen Notfall-Fallback in DayOrbitCard.vue, falls die Galerie selbst einen Fehler wirft.

2. Prioritätslogik in ActivityGallery

Die Galerie sammelt Bilder aus zwei Quellen:

const propImages = computed(() => {
  const images = props.activity?.activity?.images
  return Array.isArray(images) ? images : []
})

const localImages = computed(() =>
  Array.isArray(enrichedImages.value) ? enrichedImages.value : []
)

propImages kommen aus dem zentralen Trip-State. localImages kommen aus einem lokalen Galerie-Request.

if (images.length) {
  return images.map((image, index) => ({
    key: `${image.url}-${index}`,
    image: fallback,
    realImage: image,
    fallbackImage: fallback
  }))
}

return [{
  key: `${props.activity.id}-fallback-${categoryName.value}`,
  realImage: null,
  fallbackImage: fallback
}]

Ein echtes Bild gewinnt also nur, solange realImages.length > 0 ist. Wird diese Liste wieder leer, wird ein Platzhalter-Slide gebaut.

3. Erfolgreich geladene Bilder

Das sichtbare echte Bild registriert seinen Ladeerfolg mit:

@load="markImageLoaded(slide.realImage.url)"
@error="markImageFailed(slide.realImage.url)"

Der Erfolg wird pro URL gespeichert:

const markImageLoaded = (url: string) => {
  if (!loadedImageUrls.value.includes(url)) {
    loadedImageUrls.value = [...loadedImageUrls.value, url]
  }
  imageLoadStatus.value = 'ready'
}

Diese URL ist meistens eine interne Proxy-URL wie /activities/{activityId}/images/{imageId}/media. Der Zustand lebt aber nur in der aktuellen Galerie-Instanz. Wird die Galerie neu gemountet, sind loadedImageUrls und enrichedImages wieder leer.

4. Fehlgeschlagene Bilder

Fehler werden ebenfalls pro URL gespeichert:

const markImageFailed = (url: string) => {
  if (loadedImageUrls.value.includes(url)) return
  if (!failedImageUrls.value.includes(url)) {
    failedImageUrls.value = [...failedImageUrls.value, url]
  }
}

Innerhalb derselben Galerie-Instanz kann ein bereits erfolgreich geladenes Bild nicht mehr als failed gelten. Nach einem Remount ist dieser Schutz aber weg, weil loadedImageUrls neu startet.

5. Wo neue Bilder in den zentralen Trip-State geschrieben werden

Der zentrale Merge passiert im Workspace:

C:\Users\finnw\Documents\Reiseplaner\frontend\composables\useTripWorkspace.ts 
const updateActivityImages = (activityId: number, images: ActivityImage[]) => {
  if (!trip.value) return
  if (!images.length) return

  for (const day of trip.value.days) {
    for (const item of day.activities) {
      if (item.activity.id === activityId) {
        item.activity.images = mergeImages(item.activity.images, images)
      }
    }
  }
}

Wichtig: ActivityGallery.vue schreibt seine lokal geladenen Bilder aktuell nicht zuverlässig in diesen zentralen Trip-State. Dort steht nur:

enrichedImages.value = incomingImages

Das erklärt, warum Bilder kurz sichtbar sein können, aber nach einem Remount verschwinden.

6. Hintergrund-Preloader

Der Preloader lädt den aktiven Tag und zwei Folgetage:

const preloadUpcomingImages = async (activeIndex = 0, daysAhead = 2) => {
  const end = Math.min(currentTrip.days.length - 1, start + daysAhead)
  const activities = currentTrip.days
    .slice(start, end + 1)
    .flatMap(day => day.activities || [])
}

Eine Aktivität gilt als erfolgreich, wenn mindestens ein Bild vorgeladen werden konnte:

const warmed = await warmActivityImages(safeImages)
if (warmed) {
  imagePreloadDone.add(activityId)
  setImageStatus(activityId, 'ready')
}

Der Preloader-Status wird aktuell aber nicht direkt für die Galerieentscheidung genutzt. Die Galerie entscheidet anhand von activity.images, enrichedImages und failedImageUrls.

7. Doppelte Bildlade-Mechanismen

Es gibt zwei konkurrierende Wege:

  1. ActivityGallery.vue ruft beim Mount loadImages() auf.
  2. useTripWorkspace.ts ruft preloadUpcomingImages() auf.

Auf der Trip-Seite wird der Preloader außerdem mehrfach gestartet:

syncDayFromRoute()
workspace.preloadUpcomingImages(activeIndex.value)

Dadurch können Galerie und Preloader denselben Backend-Endpunkt fast gleichzeitig treffen.

8. Watcher und Remounts

Beim Aktivitätswechsel wird der lokale Galerie-State gelöscht:

watch(() => props.activity.id, () => {
  activeSlide.value = 0
  enrichedImages.value = null
  failedImageUrls.value = []
  loadedImageUrls.value = []
  imageLoadStatus.value = 'idle'
  loadImages()
})

Zusätzlich wird die Galerie über diesen Key neu erzeugt:

:key="(selectedActivity || day.activities[0])?.id"

Das ist aus UI-Sicht nachvollziehbar, aber gefährlich, wenn echte Bilder nur lokal in der Galerie gespeichert sind.

9. CSS- und Rendering-Logik

Aktuell liegt das echte Bild über dem Platzhalter:

.gallery-fallback-image {
  z-index: 1;
  opacity: 1;
}

.gallery-real-image {
  z-index: 2;
  opacity: 1;
}

Im aktuellen Stand sieht CSS nicht nach der Hauptursache aus. Der Platzhalter wird sichtbar, wenn das echte Bild nicht mehr im DOM ist oder als failed gilt.

10. Backend-Endpunkt für Bild-Enrichment

Der Endpunkt ist:

C:\Users\finnw\Documents\Reiseplaner\backend\src\main\java\de\travelmate\activity\ActivityResource.java 
@POST
@Path("/{id}/images")
public List<ActivityImageDto> enrichImages(@PathParam("id") Long id) {
    return imageService.ensureImages(id);
}

Der Service ist grundsätzlich idempotent:

if (!activity.images.isEmpty()) {
    return activity.images.stream().map(ActivityImageDto::from).toList();
}

Restrisiko: Bei parallelen Requests können zwei Requests gleichzeitig sehen, dass noch keine Bilder existieren. Wenn einer beim Speichern wegen Parallelität scheitert, landet er im Catch und gibt leer zurück:

} catch (RuntimeException exception) {
    LOG.warnf(...)
    return List.of();
}

11. Interne Media-URL

Das Frontend bekommt meist keine direkte Google-URL, sondern eine interne Proxy-URL:

String url = image.url == null || image.url.isBlank()
    ? "/activities/" + image.activity.id + "/images/" + image.id + "/media"
    : image.url;

Der Proxy löst diese URL bei jedem Bildaufruf über Google auf:

return Response.temporaryRedirect(URI.create(url)).build();

Mögliche Fehlerquellen sind Google-Timeouts, 403 durch Key-Restrictions, ungültige Photo-Referenzen oder langsame Redirects. Der Backend-Timeout liegt bei etwa 3 Sekunden, der Frontend-Warmup-Timeout bei 10 Sekunden.

12. Wahrscheinlichkeitsbewertung

Ursache Wahrscheinlichkeit Begründung
Lokaler Galerie-State geht verloren Sehr hoch Galerie speichert per enrichedImages lokal. Beim Remount wird dieser Zustand gelöscht.
State-Overwrite mit alter Activity ohne Bilder Hoch replaceTrip(updated) ersetzt den ganzen Trip. Eine Antwort ohne Bilder kann vorhandene lokale Bilder verdrängen.
Doppelte Requests / Race Conditions Mittel bis hoch Galerie und Preloader laden parallel denselben Endpoint.
onError nach onLoad Mittel Innerhalb einer Instanz geschützt, nach Remount aber wieder möglich.
CSS versteckt echte Bilder Niedrig im aktuellen Stand .gallery-real-image hat aktuell opacity: 1 und höheren z-index.
Leere API-Antwort überschreibt Bilder Mittel Workspace schützt davor, Galerie lokal aber nicht vollständig.

13. Minimaler Fix-Vorschlag

  1. Ein Besitzer für Bilddaten: useTripWorkspace sollte die zentrale Wahrheit für echte Bilder sein.
  2. Galerie-Ergebnisse zentral mergen: Wenn ActivityGallery Bilder selbst lädt, muss sie diese in den zentralen Trip-State zurückgeben oder der Galerie-Request wird entfernt.
  3. Trip-Merge schützen: replaceTrip(updated) darf vorhandene echte Bilder nicht durch images: [] ersetzen.
  4. Backend-Parallelität absichern: Nach einem Enrichment-Fehler sollte der Service vorhandene Bilder erneut aus der DB lesen, statt direkt leer zurückzugeben.
  5. Anzeige-Regel hart machen: Wenn echte Bilder vorhanden sind, darf der Platzhalter nie den finalen Slide-State ersetzen.

Wahrscheinlichste Root Cause: Echte Bilder werden teilweise nur lokal in ActivityGallery.enrichedImages gehalten. Dieser lokale Zustand wird beim Aktivitätswechsel, Tageswechsel oder Remount gelöscht. Wenn der zentrale Trip-State keine Bilder hat oder später überschrieben wird, zeigt die Galerie wieder Platzhalter.

Risiken des Fixes

  • Wenn Bilder zu aggressiv erhalten werden, können alte Bilder länger im Client bleiben.
  • Ein explizites Entfernen von Bildern bräuchte später ein eigenes Signal.
  • Wenn Galerie und Preloader weiter parallel laden, bleiben unnötige Requests möglich.
  • Backend-Requery nach Parallelfehlern ist sicherer, aber etwas mehr DB-Arbeit.
