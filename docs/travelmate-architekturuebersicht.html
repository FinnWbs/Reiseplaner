Projektpräsentation

TravelMate
Architektur & Datenfluss

Wie aus einer Stadt, einem Zeitraum und persönlichen Interessen ein strukturierter, datenbasierter Reiseplan entsteht.

Kernidee: Das Frontend sammelt Präferenzen. Das Backend übernimmt Datenbeschaffung, Qualitätsbewertung, Planung und Speicherung.

Stack: Nuxt 3 · Java 21 / Quarkus · PostgreSQL · Docker Compose

1. Architektur in einem Satz

TravelMate nimmt Reisepräferenzen im Nuxt-Frontend entgegen, erstellt daraus über ein Java-/Quarkus-Backend einen regelbasierten Plan, reichert Orte bei Bedarf über externe Datenquellen an und speichert die Ergebnisse zentral in PostgreSQL.

3. Datenfluss: Eine neue Reise planen

  1. Der Nutzer wählt Ziel, Zeitraum, Interessen, Tempo und Tagesrhythmus.
  2. Das Frontend sendet den Erstellungswunsch an das Backend.
  3. Das Backend erstellt Reise und Reisetage und prüft den lokalen Ortsbestand.
  4. Nur falls Daten fehlen oder veraltet sind, startet ein externer Import.
  5. Die Planungslogik wählt passende, hochwertige und räumlich sinnvolle Aktivitäten.
  6. Das Backend liefert den vollständigen Reiseplan als DTO zurück; Nuxt aktualisiert Karte, Liste und Galerie.

Wie ein API-Call formuliert ist

POST /trips
Authorization: Bearer <JWT>

{
  "city": "Kopenhagen", "daysCount": 3,
  "interestIds": [1, 4, 5],
  "startDate": "2026-07-10", "endDate": "2026-07-12",
  "planningDates": ["2026-07-10", "2026-07-11", "2026-07-12"],
  "pace": "BALANCED", "dayRhythm": "BALANCED",
  "country": "Dänemark", "countryCode": "DK",
  "latitude": 55.6761, "longitude": 12.5683
}

Bei einer flexiblen Reise bleiben konkrete Start- und Enddaten leer. Stattdessen werden Tagesanzahl und optional ein Wunschmonat übertragen.

4. Was passiert im Backend mit den Daten?

1. Request entgegennehmen

TripResource nimmt den REST-Request entgegen. TripService validiert die Daten und erstellt Reise sowie Reisetage.

2. Cache-Entscheidung

ActivitySyncService prüft für Stadt und Interessen Import-Version, Aktualität, Menge und räumliche Abdeckung.

3. Orte importieren

Geoapify liefert POIs. Ein Mapper überführt Rohdaten in ein internes Kandidatenmodell: Name, Adresse, Koordinaten, Kategorien und Referenzen.

4. Wissen anreichern

Wikidata ergänzt Referenzen und Bekanntheit. Wikipedia ergänzt, wenn verfügbar, Beschreibungstexte.

5. Qualität bewerten

Ungeeignete Orte werden ausgeschlossen. Übrig bleiben Kandidaten mit Kategorie-Fit, Bekanntheit, Datenqualität und sinnvoller räumlicher Lage.

6. Plan erzeugen

Quoten, Interessen, räumliche Cluster, Vielfalt, Tageszeit und Tempo werden kombiniert. So entstehen abwechslungsreiche Tagespläne.

5. Bilddatenfluss – bewusst nachgelagert

Bilder sind kein Blocker für die Reiseerstellung. Der Plan ist sofort nutzbar; echte Bilder werden gezielt nachgeladen.

  • Der aktive Tag wird zuerst behandelt; anschließend werden die zwei folgenden Tage vorgeladen.
  • Ein gemeinsamer In-Flight-Status verhindert doppelte Bildanfragen für dieselbe Aktivität.
  • Das Backend speichert bis zu drei Google-Bildreferenzen pro Aktivität.
  • Beim tatsächlichen Rendern löst ein sicherer Backend-Endpunkt die Foto-URL auf und leitet den Browser temporär weiter.
  • Falls kein Match zustande kommt oder ein Bild temporär nicht erreichbar ist, bleibt ein Kategorie-Platzhalter sichtbar.

7. Präsentationsfazit

„TravelMate trennt bewusst zwischen schneller Nutzeroberfläche und datenintensiver Hintergrundlogik. Das Frontend sammelt Präferenzen und zeigt Ergebnisse. Das Backend entscheidet, ob Ortsdaten bereits lokal vorhanden sind, importiert bei Bedarf neue POIs, bewertet sie nach Qualität und räumlicher Eignung und baut daraus einen Tagesplan. Zusätzliche Daten wie Bilder und Highlights werden erst nachgeladen, wenn sie für die aktuelle Ansicht wirklich relevant sind.“

Die drei Kernbotschaften

  1. Schnell: Plan und Interaktion bleiben nutzbar, während Zusatzdaten im Hintergrund nachgeladen werden.
  2. Qualitätsorientiert: Nicht jeder gefundene Ort wird genutzt; TravelMate filtert, bewertet und plant regelbasiert.
  3. Effizient: Lokale Persistenz und Caching reduzieren wiederholte externe API-Anfragen.

Technische Grundlage: Nuxt 3, Java 21/Quarkus, PostgreSQL, Docker Compose. Externe Daten: Geoapify, Wikidata, Wikipedia, Google Places und OpenStreetMap-Kartenkacheln.
